The origin of HD Wallet

HD wallet is short for layered deterministic wallet(Hierarchical Deterministic). Before the emergence of HD wallet, bitcoin wallet used random numbers to generate multiple unrelated private keys. This type of wallet is called an indeterminate wallet (or random wallet).

In Nakamotoâ€™s UTXO model, the client can only use one transaction for each bitcoin address by default, so it is very inconvenient to repeatedly back up, import, and then use the key for signature transactions in the process of using the wallet.

In order to solve the problem of private key management of uncertain wallets, community members propose a method to generate a group of private keys by using â€œseedâ€ and hash function. On the basis of this idea, the key derivation(BIP32) ,Mnemonic words(BIP39),Derived path specification(BIP43/44) and other functions is added,HD Wallet, which is now unified and standardized, has been formed.

BIP32 and its friends

BIP32 is the core improvement of hierarchical deterministic wallet. It proposes a set of advanced wallet management methods based on rigorous cryptography algorithm. In the era of BIP32, there were two main types of deterministic wallets: electrum wallets and BIP32 wallets; The algorithm and design of these two kinds of wallets are similar, but BIP32 wallets eventually became the later standard.

BIP39 is the closest to users in the bitcoin HD Wallet specification. Users who have used digital wallet will be impressed by it. Do you remember the private key phrase copied by hand? It generates the seed key of the deterministic wallet through a set of easy to remember words (mnemonics).

BIP43 and BIP44 unify the understanding and use of wallet software for hierarchical path and number, which enables users to generate the same set of keys in the same blockchain with the same mnemonic words.

BIP39,BIP43 and BIP44 are all BIP32 compatible protocols, which were later used for reference by blockchains other than bitcoin, forming a consensus in the blockchain industry. This paper focuses on the details of key derivation BIP32, and other improved protocols are not introduced one by one

Characteristics of deterministic Wallet

We say that there are two intuitive characteristics of deterministic wallet:

  1. The child key can be obtained through the parent key (including: the parent private key obtains the child private key, and the parent public key obtains the child public key), but vice versa;

  2. The public key can be made public without worrying about the loss of funds.

The main scenario of publicizing this feature is in a hierarchical organization, such as the company giving the sub private key to the Department, the sub public key to the accountant and auditor, and the root private key to the financial supervisor. The almost magical function brought by this algorithm makes many people (including those who understand technology) deify the function of HD wallet. But this description of hierarchical wallet has fatal defects.

This defect is also my third level understanding of HD Wallet: Although the child private key can be safely distributed without risk to the parent private key, and the master public key can also be distributed without risk to the master private key, these two operations cannot be performed at the same time. The risk is hidden in the layered deterministic wallet algorithm. Before introducing this defect, we first study bip32 and key derivation function.

BIP32 standard

In order to more accurately introduce the definition and connotation of bip32, I suggest that readers with computer professional background read it directly BIP32 standardã€‚

Many cryptographic formulas are used in bip32 specification to define how HD wallet derives key, which is not very friendly to non computer majors. Letâ€™s put aside the rigorous definition and derivation of cryptography and describe bip32 specification in vernacular.

BIP32 mainly introduces three concepts: key derivation function, key tree and master key generation.

Child Key Derivation function

The key derived function is a specially designed function, the core part is to use the message authentication code hmac-sha512 (in essence, hash the data first, and then encrypt symmetrically). Hmac-sha512 has a feature that when you know the input data and key, you can get the output I; Only know the output I and input key, canâ€™t deduce data.

I = HMAC-SHA512(Key, Data)

The I of the parent layer decides to take the first 32 bits and the last 32 bits as the expansion private key and the encryption key respectively. The key uses the encryption key, and the data is obtained by simply splicing the expansion private key (or expansion public key) and the digital number I.

Then, we can get the conclusion that the parent private key derives the child private key, the parent public key derives the child public key, and the child private key and the child public key are a pair of keys.(Refer to the original picture below )

In addition, different from the general key derivation function, another function named â€œhardâ€ key derivation is designed. Here, the first (32-bit) of the number I is used as the identification. When the first is 0, I < 2^31, the general key derivation function is used. When the first is 1, I > = 2^31, the â€œhardâ€ key derivation function is used. For the â€œhardâ€ key derivation function, whether it is to obtain the child private key or the child public key, it needs to use the parent private key for derivation calculation. Therefore, it loses the characteristics of direct public key derivation in the general key derivation function, and becomes inflexible, and it is also safer to â€œhardâ€. The use scenario of hard key is described later.

Key tree

Because the key derivation relationship can be iterated in multiple levels, a tree key hierarchy can be constructed by using a multi-level key derivation. If it is a common key derivation, the properties of the public key derived child public key can also be transferred layer by layer. When the key derivation relationship of a certain layer is â€œhardâ€, the conduction relationship is terminated. It can be expressed as follows:

  • N(m/a/b/c) = N(m/a/b)/c = N(m/a)/b/c = N(m)/a/b/c = M/a/b/c.

  • N(m/aH/b/c) = N(m/aH/b)/c = N(m/aH)/b/c.

H stands for hard key derivation
